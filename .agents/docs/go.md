# Go Conventions

## Purpose

Use this playbook for Go code that touches persistence, generated code, tests, or repository tooling on this machine.

## When to use

- Writing or reviewing Go services, CLIs, or internal tools.
- Adding or changing SQL-backed storage.
- Introducing migrations, schema files, query files, or database access helpers.
- Reviewing generated database code and query parameters.

## Defaults

- Prefer `sqlc` for application-owned SQL. Keep schema and queries in `.sql` files and generate typed Go code from them.
- Do not hardcode vanilla SQL query strings in Go for application storage paths unless there is a narrow, documented reason.
- Use named sqlc arguments such as `sqlc.arg(name)` or named parameters in source SQL. Never use bare positional placeholders such as `?`, `$1`, or `$2` in app-authored sqlc queries when named args are available.
- Keep generated sqlc code checked in when the repo already commits generated code, and add a drift check.
- Keep handwritten Go as an adapter around generated row and parameter types. Do not leak generated types across public CLI or API boundaries unless the repo has chosen that pattern.
- Generated code does not need hand coverage, but adapter behavior must be covered by real tests.

## Workflow

### 1. Check the project shape

- If the repo already has `sqlc.yaml` or `sqlc.yml`, follow it.
- If adding SQL storage, create `internal/db/schema.sql`, `internal/db/query.sql`, and `sqlc.yaml`, or use the repo's established equivalents.
- Treat schema and query SQL as the source of truth.

### 2. Write sqlc queries with named args

Prefer:

```sql
-- name: ListJobs :many
SELECT *
FROM jobs
WHERE (CAST(sqlc.arg(filter_repo) AS integer) = 0 OR repo = sqlc.arg(repo))
LIMIT CAST(sqlc.arg(limit) AS integer);
```

Avoid:

```sql
SELECT *
FROM jobs
WHERE repo = ?
LIMIT ?;
```

### 3. Generate and adapt

- Run the repo's generation target, such as `make sqlc`, `make generate`, or the documented equivalent.
- Keep Go code focused on conversion, validation, and transaction boundaries.
- Preserve public structs and command/API behavior unless the task explicitly changes them.

### 4. Validate

- Run the sqlc drift check if present, such as `make sqlc-check`.
- Run the Go test, coverage, and build targets that match the changed package.
- For storage changes, add or adjust integration tests that exercise the real database path instead of mock SQL strings.

## Validation checklist

- Application-owned SQL lives in `.sql` files and is generated by `sqlc`.
- Source SQL uses named args, not bare positional placeholders.
- Generated code is in sync with schema and query files.
- Public Go API and CLI behavior is preserved or intentionally documented.
- Tests exercise real database reads, writes, filters, and transaction boundaries where relevant.
