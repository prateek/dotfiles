This skill is invoked in trycycle Step 9 ("Execute with trycycle-executing"). A dedicated implementation subagent is spawned and told to use this skill to implement the plan.

**What the orchestrator already handles:**
- The orchestrator spawns a dedicated implementation subagent with explicit overrides: "Do not pause between batches or wait for feedback — execute all tasks continuously. Do not ask for review."
- The orchestrator tells the subagent: "If you hit a genuine blocker (the agent cannot use its best judgment because there is no path forward, or because being wrong could cause harm), stop and report it. Do not try to work around blockers — they need human judgment."
- After implementation, the orchestrator runs its own post-implementation review loop (Step 10) with up to 8 rounds.
- The orchestrator handles finishing (Step 11) with `trycycle-finishing`.

**What to strip:**

1. **Remove the batch-pause-feedback loop entirely.** Steps 3 ("Report") and 4 ("Continue") describe pausing between batches to wait for human feedback. The orchestrator explicitly overrides this — the subagent runs continuously. Remove these steps.

2. **Remove "When to Stop and Ask for Help".** Remove the "STOP executing immediately when" section entirely.

3. **Remove "When to Revisit Earlier Steps".** The subagent doesn't revisit — it executes forward continuously.

4. **Remove Step 5 ("Complete Development") and the finishing-a-development-branch reference.** The orchestrator handles finishing. The subagent's job ends when it has implemented all tasks and committed.

5. **Remove the "Between batches: just report and wait" line** from the Remember section.

6. **Adjust batch size guidance.** Since there's no pausing between batches, the concept of "batches" becomes less relevant. Reframe: the subagent should execute all tasks sequentially, following each step exactly. Keep the concept of marking tasks in_progress/completed via TodoWrite.

7. **Do NOT frame Step 1 as "review critically"** or imply the subagent should second-guess the plan. The orchestrator already ran up to 5 rounds of plan review. Step 1 is just: read the plan and create TodoWrite entries.

8. **Make execution iterative and explicitly red/green/refactor.** The adapted skill should not read like a one-pass checklist. Within execute mode, the subagent should keep cycling through the plan's red/green/refactor work until the task is actually complete: run the failing check, make it pass, refactor/tighten the implementation and tests, and re-run the relevant automated checks after refactoring.

9. **Require full legitimate green before execution is complete.** The adapted skill must explicitly say that execution is not done merely because one targeted check turned green. The subagent should continue improving the code and the tests until all required automated checks named or implied by the plan are passing, including any broader regression suite or full project suite that the repo convention requires before completion. A failing test is usually a reason to keep working, not a blocker.

10. **Protect test quality.** The adapted skill must explicitly forbid weakening, deleting, or diluting a valid automated test just to obtain a passing result. If a test is wrong, obsolete, or lower-fidelity than a better replacement, the subagent may correct or replace it, but it must not manufacture green by making good tests less demanding.

**Critical framing requirement — binary stop/go:**

The adapted skill must have exactly two modes: execute or stop for a blocker. There is no interim state — no "document concerns," no "review critically," no "questions about the plan." The subagent proceeds unless it hits a blocker it cannot work around.

Blocker means the agent cannot use its best judgment because there is no path forward, or because being wrong could cause harm. If something is unfixably broken (system unstable, corrupted instructions, dirty file that cannot be handled properly while proceeding), the subagent MUST stop rather than silently produce broken output. But "I have concerns about the approach" is not a blocker — the plan is already reviewed.

The blocker handling section should make this clear: the only reason to stop is a blocker (the agent cannot use its best judgment because there is no path forward, or because being wrong could cause harm). Everything else is proceed. In particular, red tests and review findings are inputs to keep iterating unless the environment itself is broken.

**Resulting structure should be roughly:**

- Step 1: Load Plan (read plan, create TodoWrite entries for all tasks)
- Step 2: Execute All Tasks (for each task: mark in_progress, follow steps, work red/green/refactor, run all required verifications, mark completed)
- Blockers (the ONLY reason to stop — the agent cannot use its best judgment because there is no path forward, or because being wrong could cause harm; detecting these is critical)
- Remember list (trimmed)
