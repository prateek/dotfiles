This skill is invoked in trycycle Step 6 ("Plan with trycycle-planning"). A dedicated planning subagent is spawned and told to use this skill to produce an implementation plan.

**What the orchestrator already handles:**
- The orchestrator spawns a dedicated subagent for planning. The subagent is told not to spawn additional subagents.
- After the initial plan is written, the orchestrator may spawn fresh planning subagents for plan-editor rounds. Each fresh subagent receives the same task input plus the current plan path and must either declare the plan already excellent unchanged or improve it directly.
- The orchestrator wraps this skill with separate prompt templates for initial planning and plan-editor rounds. Keep those wrappers distinct; they share context, but their jobs are meaningfully different.
- In the plan-editor loop, each fresh planning subagent must either leave the plan already excellent unchanged or make it excellent itself. If it is not deemed already excellent, the loop continues with a fresh plan-editor cycle.

**What to strip:**

1. **Remove brainstorming preconditions.** The line "This should be run in a dedicated worktree (created by brainstorming skill)" should change to something like "This should be run in an isolated implementation workspace" (drop the brainstorming reference). The orchestrator prepares that workspace in Step 4, either as a dedicated worktree or as an already-isolated workspace in `--no-worktree` mode.

2. **Remove the plan header line about executing-plans.** The template header says `> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.` — change this to reference `trycycle-executing`.

3. **Keep the TDD emphasis, but adapt the test-step template to red/green/refactor.** The bite-sized task granularity, the plan document header format, and the "Remember" list are all core operational content the planning subagent needs. Keep the red/green/refactor philosophy, exact file paths, and complete code in plan. But do not hard-code "write a new failing test" as the default first move for every task. The subskill should teach: use high-value existing automated checks when they already cover the relevant behavior; extend them when that is the clearest path; write new failing tests when coverage is missing. After the initial green step, the task template should explicitly include a refactor/tightening step and re-run the relevant automated checks. If the task or problem statement already names automated checks that are red and need to go green, those checks must appear explicitly in the plan.

4. **Make "done" mean full legitimate green, not local spot-check green.** Plans should not stop at one targeted test passing. They should explicitly tell the implementation subagent to keep improving the code and the tests iteratively until all required automated checks for the work are green, including any broader regression suite or full project suite that the repo convention requires before completion. Make it explicit that execution is only done when the code is genuinely correct and the required automated tests pass for legitimate reasons.

5. **Protect test quality.** The adapted skill must explicitly forbid weakening, deleting, or diluting a valid test merely to obtain a passing result. Test changes are allowed when a test is wrong, obsolete, or can be replaced by a stronger or more faithful check, but "make the test easier so it passes" is not acceptable.

**Resulting structure should be roughly:**
- Overview (plan writing guidance)
- Context note (run in isolated implementation workspace)
- Save plans to `docs/plans/YYYY-MM-DD-<feature-name>.md`
- Bite-sized task granularity rules
- Plan document header template
- Task structure template with red/green/refactor steps that can either run existing high-value tests or write new failing tests, then re-run the relevant checks and broader required suite
- Remember list
